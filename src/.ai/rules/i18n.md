# Internationalization Rules

## Standard

- Use the `@nuxtjs/i18n` module. Do not add another i18n library.
- Default locale is English (`en`); it is currently the only locale.
- URL strategy `prefix_except_default`: the default locale has no path prefix, other locales are
  prefixed (e.g. `/de/...`). Adding a language must not change the default-locale URLs.
- Messages are authored in **TypeScript**, one namespace per file, aggregated by a per-locale
  `index.ts`. Register a single file per locale (`file: 'en/index.ts'`).

## File layout

```text
i18n/locales/
└── en/
    ├── index.ts        # aggregator: imports every namespace, exports the merged map
    ├── common.ts       # namespace: generic reusable labels
    ├── product.ts      # namespace: product domain
    └── ...
```

- Each namespace file default-exports a **flat object** of that namespace's keys.
- `index.ts` imports them and exports `{ common, product, ... }`. The object keys are the namespaces,
  so keys read as `t('common.appName')`, `t('product.addToCart')`.
- The aggregator (not the file name) provides the namespace; keep each export key equal to its file
  name.
- Register namespaces **explicitly** in `index.ts` (one `import` + one entry in the exported map). Do
  **not** auto-collect with `import.meta.glob` — globbing makes the merged type `unknown` and loses
  typed `t()` keys.

## Authoring copy

- All user-facing text goes through i18n (`$t` / `useI18n().t`). Never hardcode display strings in
  templates, components, or composables.
- English is the source of truth: every key exists under `i18n/locales/en/` first.
- Do not put logic, HTML, or provider data in translation values. Use vue-i18n interpolation and
  pluralization for variables and counts instead of string concatenation.

## Naming

- Namespace / file: a domain/feature, **lowercase, single word** where possible: `common`, `product`,
  `cart`, `checkout`, `account`, `auth`, `search`, `validation`, `error`. Generic reusable labels
  (save, cancel, close, loading, retry) live in `common`.
- Keys: `camelCase`, dot-separated for hierarchy (`product.addToCart`, `common.actions.save`).
- Name by **meaning, not the English text**: `cart.empty`, not `cart.yourCartIsEmpty` — copy can change
  without renaming the key.
- No abbreviations, no UI-position names (`topButton`), no sentences as keys.
- Variables via named interpolation: `summary: 'Total: {total}'` → `t('cart.summary', { total })`.
- Counts via pluralization: `items: 'no items | 1 item | {count} items'` → `t('cart.items', count)`.
- Keys are identical across every locale.

## Adding a namespace

1. Create `i18n/locales/en/<namespace>.ts` exporting a flat object.
2. Import it in `i18n/locales/en/index.ts` and add it to the exported map — **no `nuxt.config.ts`
   change needed**.
3. Replicate the file for every other locale.

## Adding a language

1. Register the locale in the i18n config (`code`, `language`, `file`).
2. Copy the whole `i18n/locales/en/` folder to `i18n/locales/<code>/`, translate the values, keep keys
   and file names identical.
3. Handle `hreflang`/alternates only through the module; keep canonical URLs correct.

## Type safety (optional)

Because messages are TypeScript, `t()` keys can be typed from the English schema: augment `vue-i18n`'s
`DefineLocaleMessage` with `typeof` the English aggregator so keys autocomplete and missing/renamed
keys fail typecheck. Enable after the module is installed.

## SSR and SEO

- Resolve the locale before render; localized content must be in the server-rendered HTML.
- Enable locale alternates/`hreflang` only once more than one locale exists.
